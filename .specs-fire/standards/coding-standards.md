# Coding Standards

## Overview

TypeScript extension code following strict mode. Prettier handles all formatting. No separate linter configured — TypeScript's strict compiler acts as the primary code-quality gate.

## Code Formatting

**Tool**: Prettier
**Config**: `package.json` → `npm run format` / `npm run format:check`
**Enforcement**: Run before commit; CI checks via `format:check`

### Key Settings

- **Trailing commas**: ES5
- **Single quotes**: true
- **Print width**: 100
- **Tab width**: 2

## Linting

**Tool**: TypeScript compiler (`strict: true`)
**Base Config**: `tsconfig.json`
**Strictness**: strict

### Key Rules

- `strict: true` — enables strictNullChecks, noImplicitAny, etc.
- `noUnusedLocals` — flag unused variables
- `noUnusedParameters` — flag unused parameters

## Naming Conventions

### Variables and Functions

| Element | Convention | Example |
|---------|------------|---------|
| Variables | camelCase | `favoriteThemes` |
| Functions | camelCase | `getFavorites()` |
| Classes | PascalCase | `FavoritesProvider` |
| Interfaces | PascalCase | `ThemeEntry` |
| Enums | PascalCase | `ThemeType` |
| Constants | UPPER_SNAKE or camelCase | `FAVORITES_KEY` |
| Private fields | camelCase with `_` prefix | `_favorites` |

### Files and Folders

- **Source files**: camelCase (e.g., `favoritesProvider.ts`)
- **Test files**: camelCase + `.test.ts` suffix (e.g., `favoritesProvider.test.ts`)
- **Folders**: camelCase (e.g., `src/tests/`)

## File Organization

### Project Structure

```
vscode-theme-favorites/
├── src/
│   ├── extension.ts          # Activation entry point

│   ├── favoritesUtils.ts     # Utility functions for favorites management
│   ├── favoritesWebviewProvider.ts
│   ├── themesProvider.ts     # TreeView provider for all themes
│   ├── themesWebviewProvider.ts
│   └── tests/                # Test files

├── resources/                # Icons and static assets
├── images/                   # Extension marketplace images
├── package.json
└── tsconfig.json
```

### Conventions

- **Provider classes**: One class per file, named after the class
- **Utilities**: Grouped by domain in `*Utils.ts` files
- **Entry point**: `extension.ts` registers all commands and providers

## Import Order

```typescript
// 1. Node.js built-ins
import * as path from 'path';
// 2. VS Code API
import * as vscode from 'vscode';
// 3. Local imports
import { FavoritesProvider } from './favoritesProvider';
```

**Rules**:
- Node built-ins first
- VS Code API second
- Local imports last
- Use `import * as` for `vscode` and Node modules

## Error Handling

### Pattern

**Approach**: Surface errors via `vscode.window.showErrorMessage`; use `try/catch` at command boundaries.

### Guidelines

- Wrap top-level command handlers in try/catch
- Never swallow errors silently
- Use VS Code's notification API for user-facing errors
- Log errors to the extension output channel when available

### Example

```typescript
try {
  await doSomething();
} catch (err) {
  vscode.window.showErrorMessage(`Theme Favorites: ${err instanceof Error ? err.message : String(err)}`);
}
```

## Logging

**Tool**: `vscode.window.createOutputChannel` (extension output channel)
**Format**: Plain text

### Log Levels

| Level | Usage |
|-------|-------|
| Info | Normal operations, state changes |
| Error | Unrecoverable failures surfaced to user |

### Guidelines

**Always log**:
- Extension activation / deactivation
- Configuration changes

**Never log**:
- Sensitive user data
- Full theme lists on every refresh (too verbose)

## Comments and Documentation

### When to Comment

- When the WHY is non-obvious (VS Code API quirk, workaround for specific bug)
- When a subtle invariant exists
- Never describe WHAT the code does — names should do that

### Documentation Format

**Functions**: One-line JSDoc only when the function signature isn't self-explanatory
**Classes**: No class-level docblocks unless the class has a non-obvious purpose
