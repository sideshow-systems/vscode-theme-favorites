# Testing Standards

## Overview

VS Code extension tests using the VS Code test runner integration. Tests live in `src/tests/` and run inside a VS Code extension host.

## Testing Framework

**Framework**: VS Code Extension Test Runner (`vscode` test shim)
**Runner**: `vscode/bin/test` (via `@types/vscode`)

## Test Types

| Type | Tool | Location | When to Use |
|------|------|----------|-------------|
| Unit | VS Code test runner | `src/tests/*.test.ts` | Pure logic, utilities |
| Integration | VS Code extension host | `src/tests/*.test.ts` | Commands, providers, configuration |

## Coverage Requirements

**Target**: Not formally enforced yet
**Enforcement**: Manual review

**Critical paths that MUST have coverage:**
- Favorites add/remove/reorder logic (`favoritesUtils.ts`)
- Theme detection and categorization
- Command handlers in `extension.ts`

## Test Naming

**Pattern**: `describe('ComponentName', () => { it('should <action> when <condition>', ...) })`

**Examples**:
- `it('should add a theme to favorites')` — happy path
- `it('should not add duplicate favorites')` — edge case
- `it('should remove theme from favorites when it no longer exists')` — defensive

## Mock Strategy

**Approach**: Use VS Code's extension host for integration; mock `vscode.workspace.getConfiguration` for unit tests where needed.

**Guidelines**:
- Prefer integration tests over mocks for VS Code API calls
- Mock only at system boundaries (configuration, file system)
- Keep mocks minimal and explicit

## Test Data

**Strategy**: Inline fixtures — small, self-contained test data defined in each test file.

**Guidelines**:
- No shared mutable state between tests
- Use `setup`/`teardown` to reset VS Code configuration after each test
